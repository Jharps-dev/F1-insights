Project plan for an F1 insights system
Executive summary
This project plan delivers a buildable path to an F1 insights system that supports (a) ingestion from
reputable community/official artefacts, (b) a canonical schema + replay hub, (c) a premium
(“swanky”) web UI with an optional desktop wrapper, and (d) an AI/analytics layer that works
deterministically on replayed sessions.


The plan is deliberately replay-first because live F1 timing ecosystems are operationally and legally
volatile. A concrete example is the documented shift in what data is “freely published” on the live feed
for some community clients since the 2025 Dutch Grand Prix, which is exactly the kind of upstream
change that can derail a live-first build. 1


Legally, a commercial direction is a gating constraint, not a “later” feature. Formula 1 2 ’s official
guidelines say that “Statistics, Results and Timing Data” are protected and that substantial
reproduction/commercial use via scraping is not permitted; its guidelines also explicitly restrict use of
“FORMULA 1 Rights” for text-and-data mining and in/with AI without an express licence. 3 The F1 TV
subscription terms (US 2024 example) also include an explicit prohibition on text/data mining and web
scraping. 4


Accordingly, the MVP is defined as a demo/replay product that uses: - OpenF1 (2023+ historical, free)
for core tracks, telemetry and events, licensed CC BY‑NC‑SA 4.0, with clear sampling-rate and delay
disclosures. 5
- FastF1 for post-session processing/caching and to record raw live timing streams for later replay
processing (not real-time use). 6
- Jolpica (Ergast-compatible) for schedules/results metadata under explicit non-commercial terms and
published rate limits (4 req/s burst, 500 req/hour sustained unauth). 7
- FIA 8 PDF documents for authoritative classifications/decisions (excellent for a “Documents” pane
and for validation). 9


A desktop wrapper is optional for MVP. If chosen, this plan recommends an SPA-first UI with optional
Tauri packaging: Tauri uses the system WebView (WKWebView / WebView2 / WebKitGTK, etc.) and
provides a security model anchored by CSP/capabilities/permissions. 10 Electron remains viable if you
want a consistent bundled runtime; Electron explicitly bundles Chromium and Node.js, and its security
checklist (CSP, context isolation, sandboxing) must be treated as non-negotiable. 11

Scope

In scope (MVP + foundations): - Data importers (OpenF1, Jolpica, FIA PDFs, FastF1 exports/recordings).

- Canonical schema and mapping layer (event log + state projections).
- Replay hub (“session player”): deterministic clock, seek/scrub, WebSocket streaming to clients.
- Premium UI: timing tower, driver inspector, telemetry chart canvas, stint/pit strategy views, race

- Analytics V1: derived metrics and heuristics (stint pace, deltas, pit windows), plus a thin AI layer for

Explicitly out-of-scope for MVP unless licensed: - Direct use of F1 digital product rights/content beyond
what is permitted (especially “substantial” timing data reuse in commercial contexts or AI training on
Formula 1 digital product data). 16
- Public redistribution of raw datasets from sources that are “non-commercial only” (OpenF1, Jolpica)
without ensuring licence compliance.    17




Success criteria (measurable)

Functional: - Import and replay at least three sessions end-to-end (e.g., Practice, Qualifying, Race) fully
offline from stored artefacts.
- Replay controls: play/pause, 0.5×/1×/2×/8× speed, seek-to-time, jump-to-event.
- UI parity for core components: timing tower, driver card, telemetry view, stints/pits, race control,
documents.


Quality: - Replay determinism: identical inputs → identical derived state timelines (hash checks across
runs).
- UI performance: animation budget consistent with RAIL guidance (aim for ~10ms JS per frame for
60fps class interactions) and stable telemetry rendering; avoid frame drops under typical session
payload. 18
- Accessibility: meets WCAG contrast guidance for 4.5:1 (normal text) and 3:1 (large text), keyboard
navigation for critical controls, and non-colour indicators for sector state. 19


Compliance: - In-app disclaimers and attribution for OpenF1/Jolpica licences and explicit prohibition on
commercial use without proper licensing. 20

Frontend platform decision

This plan recommends SPA-first with optional desktop packaging.


     • Tauri: system WebView approach (e.g., WKWebView, WebView2, WebKitGTK), documented
       process model and security features like CSP and capabilities/permissions. 22
     • Electron: bundles Chromium and Node.js; reasoned trade-off is consistent runtime vs package
       size, with an explicit security checklist (CSP, context isolation, sandboxing).   11



Decision rule: - If the demo must be offline-first and distributed as an installer with dataset bundles →
include Tauri packaging in MVP. 23
- If runtime consistency across all user machines is paramount and you can accept a larger bundle →
consider Electron, but adopt best practices from day one. 24


Data-provider integration plan (demo pathways)

OpenF1 (core demo feed) - Scope: telemetry, laps, weather, race control, positions (2023+). OpenF1
documents a 3.7 Hz telemetry sampling rate and ~3-second live delay (for live contexts), plus JSON/CSV
support and 2023+ coverage. 5
- Access: REST for historical; authenticated streaming via MQTT/WebSocket requires OAuth2 token
workflow. 25

- Licence: CC BY‑NC‑SA 4.0; stated as unofficial and intended for educational/research/non-commercial
fan engagement. 26


FastF1 (post-session and replay-friendly) - Scope: transforms timing/telemetry into analysis-friendly
dataframes, with caching/rate limit system. 27
- Critical constraint: its live timing client saves raw messages and explicitly says it is not possible to use
this data during a session; processing happens after session load. 28


Jolpica (historical metadata) - Scope: schedules/results/standings style metadata, Ergast-compatible
base path. 29
- Rate limits: unauth burst 4 req/s, sustained 500 req/hour. 30
- Terms: non-commercial use; CC BY‑NC‑SA 4.0; volunteer-run with no guarantee of correctness/uptime.
 7




FIA PDFs (official documents) - Scope: authoritative classifications/decisions and a strong “Documents”
UI module. The FIA’s decision documents pages list documents by season/event. 9


Optional licensed provider (commercial track) - If commercialisation is a target, plan a parallel
procurement workstream. Example: Sportradar 31 markets official F1 data and low-latency feeds
(including gRPC event stream documentation for F1 live data). 32
- This is a contract/SLA and integration effort; treat it as a milestone with gating criteria (see below).


Canonical schema and mapping tasks

Canonical   schema     (v1)   must   support:   -   Append-only    event   log   with    event_time_utc ,
ingest_time_utc , session_id , driver_id , kind , payload . - Versioning and backward-
compatible decoding (schema evolution). - Dual-time handling: source event time + replay time (for
deterministic clock).


Mapping tasks: - Identity resolution: unify driver_number , driver_code , team naming across
feeds; Jolpica can seed canonical driver/team IDs and schedule keys. 29
- Time alignment: OpenF1 uses timestamped records; FastF1 often includes session-relative time
constructs and has specific time explanations to respect (convert reliably). 33
- Cross-validation: compare derived classification vs FIA PDFs where feasible (post-session). 9

Legal/licensing gating and procurement steps

This is a formal stage, not an informal review, because Formula 1 guidelines explicitly cover both timing
data scraping/commercial reuse and AI/text-data mining restrictions. 3


Gating checklist: - Confirm MVP is non-commercial demo and datasets are not redistributed in ways
incompatible with CC BY‑NC‑SA terms (OpenF1 and Jolpica). 17
- Ensure the app does not claim affiliation; display trademark and source disclaimers (both OpenF1 and
FastF1 present similar disclaimers in their materials). 43
- If commercial: engage licensing counsel, then either: - Seek express licence from Formula 1 companies
for data/AI usage, or - Procure a licensed provider feed and integrate under contract (e.g., Sportradar F1
live feed docs describe a gRPC event stream and position it as low-latency). 44


Procurement workstream (optional, parallel): - Requirements definition (latency, fields, SLAs,
redistribution rights). - Vendor evaluation and data-field gap analysis vs canonical schema. - Contract +
privacy/security review. - Integration sprint(s) once credentials and sandbox endpoints available.


QA and testing strategy

Replay determinism: - For each dataset, generate: - event_log_hash (hash of canonical JSONL) -
state_timeline_hash (hash of projected state at fixed ticks) - CI: run determinism tests on every PR
(small dataset) and nightly (full dataset).


Load and performance: - Rendering: ensure stable animation time budgets; web performance guidance
implies 60fps class interactions need strict per-frame budgets (16.7ms total, ~10ms JS recommended).
 18

- Backend: WebSocket fan-out tested at target concurrency (define a demo concurrency number; e.g.,
50 clients) with CPU/memory headroom.


Accessibility: - Contrast thresholds (4.5:1 for normal text; 3:1 for large text) and ensure sector state isn’t
colour-only. 19




                                                      9
Performance targets and budgets

UI targets (MVP): - Timing tower updates: 2–4 Hz “state tick” (fast enough for demo, reduces DOM
churn). - Telemetry plots: render loop driven by requestAnimationFrame() , with decimation and
worker offload; rAF schedules work according to browser frame timing.     45

- Per-frame budget: aim for ~10ms JS on hot paths (RAIL). 46


Backend targets (MVP): - Replay hub: <50ms state tick generation at 4Hz for a full grid dataset on a
typical dev laptop. - WebSocket publish: <100ms end-to-end within local network for demo.


Security checklist (web + optional desktop)

Web: - CSP, dependency pinning, SRI/lockfiles, threat model for WebSocket events (validate schema;
treat server as authority). - Sanitize any HTML/text rendering from PDFs or feeds; avoid untrusted HTML
insertion.


Electron (if chosen): - Enforce CSP; enable context isolation; enable sandboxing; never expose full Node
APIs to renderer. 47


Tauri (if chosen): - CSP configured; minimise capabilities/permissions; explicitly allowlist only required
commands and windows. 48


Monitoring/observability: - Import success rates, 429/backoff counts (Jolpica explicitly states 429
behaviour). 30
- Data completeness metrics per “kind” (telemetry %, lap count, etc.). - UI telemetry: frame time
histograms, dropped frames, memory.

Import script outlines (no full code)

Node importer outline (OpenF1) - Inputs: session_key , optional driver_numbers[] , date range. -
Steps: - For each endpoint ( car_data , laps , weather , race_control , etc.), fetch paginated/
filtered results. - Cache responses by URL+params hash. - On 429/5xx: exponential backoff with jitter. -
Map each row into canonical event (source, kind, event_time, payload). - Write JSONL event log; build
secondary indexes (time → offset) for fast seek. - Acceptance tests: - Event log sorts deterministically by
(event_time, source_priority, stable tie-break). - Round-trip decode and schema validation.


Python exporter outline (FastF1 + cache) - Use FastF1 cache configuration and offline mode for
deterministic reuse; cache hits don’t count to limits and caching is enabled by default. 55
- Steps: - Cache.enable_cache(path) early. - Load session, export standard frames (laps, telemetry
curves, results). - Emit canonical event forms or store as “parquet sidecar” referenced from manifest. -
Risks: upstream availability constraints; avoid live reliance in MVP.



                                                    12
Jolpica importer outline - Enforce published limits (4 req/s burst; 500/hr sustained unauth). 30
- Prefer aggregated endpoints and offsets; avoid per-lap loops. - Map schedule/results into canonical
metadata tables (drivers, constructors, events).
