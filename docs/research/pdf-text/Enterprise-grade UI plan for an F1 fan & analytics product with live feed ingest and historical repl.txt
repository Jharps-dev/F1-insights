Enterprise-grade UI plan for an F1 fan &
analytics product with live feed ingest and
historical replay
Executive summary
A premium, enterprise-grade F1 fan & analytics UI is feasible with a web-first SPA that consumes a
canonical event stream (live + replay) over WebSockets, with an optional desktop shell (Tauri
preferred for footprint/security controls) for offline dataset bundling and “replay-first” demos. The best
near-term open/community backbone is OpenF1 for historical (free) and live (paid) timing/telemetry/
race-control data over REST plus MQTT/WebSocket streaming, with explicit documented auth and topic
conventions. 1


Key architectural choices that keep latency low and UI “swanky”:


1) Treat the UI as a subscriber to a canonical timeline, not as a direct consumer of multiple upstream
formats. Build a replay clock and a WebSocket gateway that emits render-ready event envelopes
(idempotent, ordered, backpressure-aware). OpenF1’s streaming messages already include an ever-
increasing _id and a stable _key that can be leveraged for ordering and upserts. 2
2) Use a performance budget discipline: keep animation work inside a ~10ms/frame budget for 60fps
experiences and handle input within ~50ms where possible; when that fails, degrade gracefully via
decimation and progressive rendering. 3
3) Enterprise readiness from day one: adopt OAuth2/OIDC SSO patterns and RBAC, ship full audit
logging, add strict CSP/desktop IPC hardening, and implement retention/minimisation controls aligned
to UK GDPR principles. 4
4) Licensing is a gating item: Open/community sources are typically non-commercial (e.g., CC
BY‑NC‑SA) and official Formula 1 properties explicitly restrict scraping, substantial reuse, and AI/text-
data-mining uses without an express licence. Your MVP should therefore be framed as demo /
research / non-commercial unless you procure a licensed feed. 5


Two build tracks are recommended:


     • Enterprise MVP (12–20 weeks, target ~16 weeks): Web SPA + backend replay hub + OpenF1
       import + “swanky” core components (timing tower, track map, telemetry canvas, stints/pit, race-
       control feed, driver/highlight cards) + observability and security baseline.
     • Post‑MVP (licensing + live SLA): add licensed low-latency provider (e.g., Sportradar gRPC event
       stream) for production live analytics; harden multi-tenant controls; expand AI analytics features
       and evaluation.

Assumptions and constraints
Assumptions (explicit):


     • Target OS, budget, and expected concurrency are unspecified; the plan favours a web-first
       approach and isolates “desktop/offline” as an optional path.
     • Target latency SLA is unspecified; the system is designed to support sub-second internal
       pipelines but must still tolerate upstream multi-second delays and broadcast-delay
       synchronisation controls. OpenF1 indicates live updates can be ~3 seconds behind track events
       and highlights that it can be “faster than most TV broadcasts.” 6
     • “Enterprise-grade” implies SSO/RBAC, audit logging, monitoring/alerting, secure SDLC, and
       privacy-by-design.

Non-negotiable constraints (licensing/legal):


     • OpenF1 is licensed under CC BY‑NC‑SA 4.0 (non-commercial share-alike), and is explicitly
       positioned as an unofficial community project. 7
     • Jolpica’s Ergast-compatible API is non-commercial and CC BY‑NC‑SA 4.0, with published limits
       and disclaimers (volunteer-run; no uptime guarantees). 8
     • Formula 1 guideline language states timing/statistics/results data is protected by copyright/
       database rights, warns against commercial scraping/substantial reproduction, and includes
       explicit restrictions related to AI/text-and-data-mining without an express licence. 9
     • F1 TV subscription terms include “not use… any text or data mining or web scraping” in relation
       to the service/site.   10



Implication: MVP should use only sources whose terms you can comply with, and the UI should ship
with a clear “demo/non-commercial” disclaimer and removable branding.


Platform decision and frontend framework rationale

Recommended platform strategy

Web SPA first, desktop optional (Tauri/Electron)


     • Web SPA is the fastest path to broad reach, lowest distribution friction, and easiest A/B iteration.
       For real-time dashboards, the key is tight control of render loops and high-performance
       streaming updates—not SSR.
     • Desktop shell is valuable for:
     • bundling a replay dataset for demos/offline use,
     • offering native multi-window experiences (multi-screen timing/telemetry),
     • secure local storage of large session packs.

Desktop trade-offs are most cleanly handled by keeping your UI a pure web app and swapping only the
“host” (browser vs desktop shell).

Frontend framework choice

For an enterprise real-time analytics UI, the primary selection criteria are:


      • performance control (fine-grained update priority, rendering strategies, worker integration),
      • ecosystem maturity for enterprise testing/observability,
      • long-term maintainability (hiring, libraries, conventions).

A strong default choice is React + Vite:


      • Vite is a modern build tool with fast dev server and production builds (Rollup-based), widely
        adopted for SPAs. 17
      • React’s ecosystem has the deepest pool of charting, virtualisation, e2e testing, and enterprise
        observability integrations (practically relevant even if other frameworks are faster on micro-
        benchmarks).

When to choose Next.js instead: if you need authenticated SSR pages (marketing, account pages, SEO)

When to choose SvelteKit instead: if your team prefers smaller runtime and you want high

Flutter and Qt are valid but shift the product away from the web toolchain. Flutter supports desktop

stacks. 20 Qt is powerful but has its own licensing model that must be assessed for enterprise
distribution. 21

Responsive layouts (desktop/tablet/mobile)

A real-time motorsport dashboard is naturally “desktop-first”, but mobile support is still viable using
adaptive composition:


      • Desktop: 12/24-column grid, multi-panel (timing + map + telemetry + events).
      • Tablet: 2-column, collapsible telemetry.
      • Mobile: single column; “timing tower” becomes a virtualised list with a sticky header.

Critical: avoid reflow “storms” during live updates; prefer GPU-friendly transforms and localised DOM
updates.


Performance budgets and rendering rules

Use explicit budgets aligned to web performance guidance:


      • Animation frame budget: aim for ~10ms of app work per frame (browsers need time to
        render). 23
      • Responsiveness: provide user feedback ideally within 50–100ms. 24
      • Animation scheduling: use requestAnimationFrame() rather than timers for visual
       updates.   25



For telemetry/position streams, the UI must implement: - decimation/downsampling (server-side
preferred), - ring buffers per driver, - Web Workers for parsing/aggregation, - backpressure controls
(subscription toggles, “lite mode”).

Canonical data strategy (UI contract)

The UI should consume a single “canonical” stream:


      • Event envelope for every update (timing, lap, position, telemetry point, weather, race control,
        radio).
      • Strong ordering semantics (monotonic sequence per source; deterministic replay).
      • Idempotent upserts keyed by stable IDs.

OpenF1 already provides two crucial streaming fields: - _id (ever-increasing; chronological sort), -
_key (document identifier for updates to the same object).       36

OpenF1 (demo + live)

OpenF1 provides REST for historical queries and documents MQTT/WebSocket streaming for live
sessions, with a documented OAuth2 token endpoint and connection details. 37

      • Token: POST https://api.openf1.org/token (username/password), tokens expire in 1
        hour. 38
      • MQTT broker: mqtt.openf1.org:8883 (TLS), WebSocket: wss://mqtt.openf1.org:8084/
       mqtt .   36

      • Topics map 1:1 to REST endpoint paths (e.g., v1/location , v1/laps ).         36

      • Formats: JSON by default; CSV available via csv=true .        39

      • Sampling/latency: car telemetry and location at ~3.7 Hz, intervals/gaps updated about every 4
        seconds (documented). 40
      • Tiering/rate limits: historical free tier has published request limits; live streaming in paid
        sponsorship tier. 41

SignalR live timing (via community wrappers)

Multiple open-source projects connect to livetiming.formula1.com via SignalR. This is an official
endpoint, but that does not imply an official licence for reuse. In practice, you must treat direct
consumption as high legal/compliance risk for commercial products. Formula 1’s guidance warns
against commercial scraping/substantial reproduction. 43


Community wrapper behaviour you can reuse for demo tooling:


      • OpenF1.Data (NuGet) documents that the live timing stream is SignalR at https://
       livetiming.formula1.com/signalr and lists topics like TimingData ,
        RaceControlMessages , WeatherData , plus optional CarData.z and Position.z . It
       also describes a record+replay file format ( subscribe.txt , live.txt ) and a replay client
        that re-emits messages with calculated delay. 44
      • FastF1 includes a live timing SignalR client that records raw data; critically, it states it is not
        possible to use this data during a session, only post-session processing. 45

Replay timeline key insight: FastF1 explains that SessionTime is relative to session start, with
SessionTime zero marking the beginning of data streams, and Date providing an absolute UTC
timestamp. This is exactly the semantic split you want in your replay engine (clock-relative vs wall-clock).
 46

Security checklist (web + desktop)

Token handling: OpenF1 itself warns that exchanging username/password for OAuth2 tokens must be
done in backend code, not embedded in client-side apps; it also notes client-side token storage risks.
 62

CSP: CSP is an effective defence-in-depth against XSS and injection attacks; Electron’s own guidance




                                                 12
recommends CSP for content loaded inside Electron. 63
Desktop IPC hardening: adopt least privilege—Tauri’s permission model is explicitly designed around
scoped command access. 11


Enterprise AuthN/AuthZ patterns:


     • OAuth 2.0 is the standard authorisation framework for tokens and delegated access. 64
     • OpenID Connect adds an identity layer on top of OAuth 2.0. 65
     • RBAC is a standard access control model where permissions attach to roles rather than
       individuals. 66
       IETF 67
       NIST 68

Privacy and compliance (UK/enterprise):


     • Data minimisation and storage limitation (“keep only what you need, only as long as needed”)
       are core GDPR principles; the ICO provides guidance and emphasises there are no fixed
       retention limits—your schedule must match your purpose. 69
     • Audit logging is useful for accountability but logs must be secured and used appropriately. 70
       Information Commissioner's Office 71

Observability and instrumentation points

Use distributed tracing and structured logs from day one:


     • Memorably, Sentry describes distributed traces as transactions and spans, capturing operations
       across services. 72
     • Honeycomb describes distributed tracing as end-to-end request visibility across services, useful
       in microservice architectures. 73
     • For open-source tracing backends, Jaeger/Zipkin are typical options (selection depends on your
       existing stack); use OpenTelemetry conventions where possible.

Instrumentation points (minimum set):


     • Collector adapters: connection uptime, reconnects, message lag (event_ts → ingest_ts), decode
       errors.
     • Stream processor: per-topic throughput, watermark lag, join failures.
     • WebSocket gateway: subscriptions count, backpressure drops, per-client queue depth.
     • Replay clock: drift, determinism checksum errors.
     • Frontend: FPS, long tasks, WebSocket message-to-render latency, error boundaries.

Charting and telemetry rendering choices

Charting choices should be separated into: 1) “interactive analytic charts” (hover, zoom, multi-series
comparisons), 2) “high-frequency/continuous traces” (canvas/WebGL, ring buffers).

     • D3 modules have permissive licences (example: d3-selection licence grants broad permission).

Telemetry canvas (ring buffer + worker parsing)

Key UI principle: parsing and aggregation should not occur on the main thread. Use
 requestAnimationFrame() for draw scheduling. 25



  export function TelemetryCanvas({ streamId }: { streamId: string }) {
    const ref = React.useRef<HTMLCanvasElement | null>(null);




                                                  15
    React.useEffect(() => {
      const canvas = ref.current!;
      const worker = new Worker(new URL("./telemetry.worker.ts",
  import.meta.url));


      worker.postMessage({ type: "bind", canvas }, [canvas as any]); //
  OffscreenCanvas where supported
      worker.postMessage({ type: "subscribe", streamId });

Replay mode and offline bundling

Replay should be deterministic and distributable:


      • Data store: JSONL event logs + Parquet for columnar analytics; local SQLite/DuckDB index for
        random-access (desktop edition).
      • Replay clock: convert envelope event_ts → “replay now” based on speed (0.5x–16x), and allow
        broadcast delay offsets.
      • UI: a single “live” UI should also work in replay by switching the “clock source”.

OpenF1.Data’s documented record/replay model (subscribe snapshot + append log with calculated
delay) is a workable template. 44
