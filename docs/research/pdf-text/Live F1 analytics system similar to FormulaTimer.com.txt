Live F1 analytics system similar to
FormulaTimer.com
Executive summary
A FormulaTimer-style product is essentially a low-latency event pipeline plus a UI that can digest
frequent partial updates (lap/sector deltas, intervals, pit events, and telemetry) without collapsing
under jitter, upstream churn, or legal constraints. On the surface, FormulaTimer advertises “millisecond-
accurate lap times and sector data”, a “live leaderboard with gaps and intervals”, tyre info, team radio,
DRS tracking, and session replay. 1


In March 2026, the most realistic “live” build path for a fan/community product is to treat “live” as multi-
second latency and focus your engineering on robust ingest + replay + deterministic analytics,
because:


     • OpenF1 offers explicitly documented real-time streaming via MQTT/WebSocket (with OAuth2), a
       stated ≈3s data delay, and scene-specific refresh rates (e.g., intervals about every 4 seconds).
         2

     • The community ecosystem using the underlying live timing feed has faced upstream changes
       and partial data availability—e.g., FastF1 users reported CarData.z disappearing from live
       channels in late August 2025. 3
     • The likely “commercial-grade” route is a licensed feed such as Sportradar’s Live Data gRPC event
       stream, which is designed for low-latency integration and includes replay tooling for robust
       testing, but is a B2B contract and not meant to be called from client-side apps. 4

The biggest gating constraint for any product that looks like a live timing platform is licensing. Formula
1’s own published guidelines state that “FORMULA 1 Rights” cannot be used for text/data mining or with
AI (including LLM training/development/operation) without an express licence, and they restrict
commercial use without a licence. 5 Separately, the F1 TV subscription terms (US 2024 example)
explicitly prohibit text/data mining and web scraping of the service. 6


Assumptions (because you left them open-ended):
- Target latency SLA: sub-second internal pipeline where feasible, but external truth depends on
provider refresh rates (often 1–4s+ for many fields). 7
- Budget: MVP built first on OpenF1 streaming; commercialisation requires procurement of a licensed
feed. 8
- User base: unknown; architecture presented in tiers from “solo demo” to “multi-tenant live”.

- MQTT over WebSockets: wss://mqtt.openf1.org:8084/mqtt . 11
- Auth model: use the OAuth2 access token as the MQTT/WebSocket password; username can be any
non-empty string (or email if your library requires it). 12
- Topic model: MQTT topics correspond directly to REST endpoint paths, e.g. v1/sessions , v1/

Message format and ordering: - OpenF1 states MQTT/WebSocket messages mirror the corresponding

“update-in-place” semantics (important for laps that get refined as sectors arrive). 14


Rate limits and “live” definition (publicly documented): - OpenF1 FAQ states the free tier allows 3
req/s and 30 req/min, and supporters get double capacity. 15
- OpenF1 distinguishes “live” as data from a session in progress or ended less than 30 minutes ago;
after that it is historical and free. 15
- Stated data delay: “Real-time data typically updates about 3 seconds after live events.” 15


Implications for your latency SLA: - Your best-case low-latency pipeline can be sub-second end-to-end
internally, but “truth” freshness is bounded by provider update cadence: - Telemetry ( car_data ) ~3.7
Hz sample rate. 16
- Location ~3.7 Hz sample rate. 17
- Intervals update approximately every ~4 seconds. 18
- Weather updates every minute. 18

Low-latency ingest and streaming architecture for sub-second to
multi-second SLAs

Latency SLAs you can actually specify

Because your feed cadence can be ≥4 seconds for some data types (e.g., intervals in OpenF1), define
separate SLAs:


     • Ingest SLA (internal): time from ingest receipt → canonical event persisted + fan-out ready.
     • Target: p50 < 100 ms, p95 < 300 ms (achievable with in-memory + log append).
     • Analytics SLA: time from canonical event persisted → derived metrics published.
     • Target: p95 < 500 ms for “tower metrics”, < 1–2s for heavier models.
     • End-user freshness SLA: bounded by provider update cadence:
     • “Telemetry insights”: can refresh multiple times per second (3.7 Hz sampling). 17
     • “Intervals”: expect stepwise updates ~4s. 18
     • “Weather”: minute-level. 18

Architecture options

Option A: Direct ingest → WebSocket UI (fastest MVP, lowest ops)

     • One backend service:
     • Subscribes to OpenF1 MQTT topics
     • Normalises to canonical schema
     • Maintains in-memory session state




                                                   4
     • Pushes state deltas to browsers (WebSocket)

When to choose: - You’re building a single-tenant demo or small fan site. - You accept multi-second
cadence where the source dictates it. 7


Key security note: OpenF1 explicitly warns that client-side tokens are exposed in browser memory and
suggests the backend manage MQTT/WebSocket and stream to clients. 12


Option B: Edge collectors → Kafka/Redpanda → stream processor (scalable live analytics)

This mirrors how betting/media-grade feeds are typically integrated: you separate “collection” from
“processing”.


     • Edge collector (per region):
     • Maintains persistent connections (MQTT / SignalR / gRPC)
     • Handles reconnect/backoff and rate limits

     • Emits canonical events into a log


     • Event log: Kafka or Redpanda


     • Partition by (session_key, driver_number) or (session_key, kind)

     • Enables replay, backfill, and deterministic debugging


     • Stream processing: Flink / ksqlDB / bespoke consumers


     • Builds live tower state, stint models, win-prob updates

     • Writes to online stores (Redis/KeyDB, ClickHouse, Timescale)


     • Fan-out layer:


     • WebSocket gateway for UI clients
     • Rate limiting, per-user subscriptions, backpressure

This option aligns with Sportradar’s own guidance emphasising replay usage during integration for
robustness. 23


Option C: Dual-source ingest (OpenF1 + licensed provider) behind an adapter contract

If you intend to commercialise, build an adapter interface from day one:


     • ProviderAdapter → produces canonical events
     • SessionStateBuilder consumes only canonical events
     • AnalyticsEngine consumes only canonical events / features

This is the only clean way to start with OpenF1 (non-commercial orientation) and later replace with a
licensed provider while preserving analytics/UI. 33

Buffering, replay, and fault tolerance

A FormulaTimer-style experience benefits disproportionately from robust replay:


     • Provider-level replay: Sportradar explicitly references replay functionality and booking
       replayable stages for testing. 23
     • Community replay: A SignalR client package describes recording incoming events and replaying
       them with calculated delay; similarly, FastF1 can record raw SignalR messages for post-
       processing. 34

System design patterns: - Append-only event log (immutable, time-ordered) + idempotent
consumers. - Deterministic state projection: given the same event sequence, state must match
(critical for analytics correctness). - Graceful degradation: if telemetry topics vanish (as reported for
 CarData.z ), fall back to timing-only analytics and flag reduced fidelity. 3

Real-time analytics features and models

Concrete “live” features (what users perceive as valuable)

FormulaTimer emphasises gaps/intervals, tyre strategy, telemetry comparisons, and replay.              1

Translating that into analytics primitives:


Live timing/tower: - Live gap to leader + interval to car ahead (streaming aggregates over position/
interval feeds). 36
- Sector performance deltas: personal best vs session best; lap invalidations (needs race-control/track
status style messages). 21
- DRS activation tracking (requires an upstream field; treat as optional because not all sources expose it
cleanly). 1


Strategy: - Undercut probability and “pit now” alerts - Live stint degradation estimation - Pit-stop loss
model with safety-car sensitivity (rule-based + Bayesian adjustments)


Telemetry: - Telemetry anomaly detection (lift-and-coast onset, brake failures, sensor dropouts) - Driver
comparison overlays (normalised traces)


Outcome: - Live win probability / podium probability updates

Algorithms and models suited to streaming constraints

The key is to match model complexity to update cadence:


Streaming aggregations (always-on, low cost)

      • Interval smoothing with bounded lag (EWMA / median-of-last-N updates)
      • Mini-sector PB/SB classification
      • Pit stop detection (state machine over lap status, pit entry/exit flags if available)

These are cheap enough to run at every event tick.


Bayesian updates (robust under uncertainty)

      • Win probability can be framed as Bayesian updating on latent pace and incident risk, updating
        posterior as new intervals/lap times arrive.
      • Useful because live feeds are noisy and stepwise (especially 4s interval updates). 18

Survival/hazard models (incidents, safety car likelihood)

      • Model hazard of SC/VSC or DNFs as a function of track status, weather, and historical rates.
      • Online inference uses pre-fit parameters; runtime is light.

Lightweight ML for strategy triggers

      • Gradient boosted trees / logistic regression for “undercut success probability” given:
      • Δpace estimate, tyre age/compound, gap, pit lane loss, traffic density, track evolution proxy

The focus is deterministic, interpretable alerts rather than black-box deep models.


Feature engineering for streaming telemetry and timing

From OpenF1, you can derive high-quality streaming features because telemetry/location are ~3.7 Hz:
 17




Examples (computed per driver, per sliding window): - Speed mean/max, throttle/brake duty cycle,
braking points consistency - Time-to-apex proxy: correlate speed troughs with location progress - “Pace
residual”: actual lap/mini-sector vs expected from stint model - “Traffic exposure”: variance of interval +
proximity events


Data-model alignment: - Use _id and _key (OpenF1 streaming) to handle “updates to the same lap
object” without double-counting. 37


Training data and evaluation

For live analytics you need two tracks:


      • Offline training/evaluation on historical sessions:
      • OpenF1 provides historical data from 2023 onwards and clearly differentiates live vs historical.
        38




                                                       7
     • FastF1 is widely used to build post-session datasets and has caching/rate limit controls to
       support repeatable experiments. 39


     • Online evaluation during live sessions:


     • Log predictions and realised outcomes (e.g., undercut alerts: true/false)
     • Compute calibration (Brier score) for probabilities and MAE/RMSE for interval forecasts

Real-time inference constraints: - Prefer CPU-friendly models; avoid heavy GPU dependencies in the
MVP. - Batch inference per “tick” (e.g., every second), not per-message, to control tail latency. - Treat
“model serving” as a low-latency microservice only when you exceed the capacity of in-process
inference.

Why the collector layer matters: - It isolates your system from transport quirks: MQTT topic fan-out,
SignalR negotiate flows, gRPC stream restarts. - It is where you implement reconnect/backoff and
source-specific idempotency. - It is where you enforce “don’t bring provider tokens into the
browser” (OpenF1 explicitly suggests backend-managed streaming as more secure). 12


Frontend real-time UI patterns

A FormulaTimer-style UI is “swanky” when it feels fast and stable even when the feed isn’t.


Patterns that make streaming UIs feel premium: - Progressive updates: show partial lap objects
immediately; refine as sector splits arrive (requires update semantics like OpenF1’s _key concept). 37
- Backpressure-aware subscriptions: allow users to subscribe to fewer topics or lower-frequency
aggregates (tower tick rate vs raw telemetry).
- Optimistic UI only for user actions: replay speed changes, selected driver, column layout—never for
data truth.
- Ring buffers for telemetry: last N seconds stored per driver, decimated for rendering.
- Graceful degradation states: if telemetry disappears (e.g., CarData.z missing), show a “Timing-
only mode” badge.   3




Performance envelope: - If you aim for 60fps-class smoothness, strict per-frame budgets are necessary
(RAIL suggests aiming for ~10ms of JS work per frame for responsive animation). 25


Security considerations (live systems)

     • Token handling:
     • OpenF1 notes that client-side tokens sit in browser memory and suggests a backend-managed
       streaming bridge for better security. 12

     • Sportradar indicates server-to-server integration; do not call B2B feeds unmediated from clients.
        40




     • Legal constraints:


     • Formula 1 guidelines explicitly prohibit use of “FORMULA 1 Rights” for text/data mining or with
       AI without express licence, and restrict commercial use without licence. 5
     • F1 TV terms explicitly prohibit text/data mining and web scraping. 6
     • If you want to run AI features at scale commercially, the procurement path is not optional.

Procurement/licensing steps for commercial live feeds

If you want to monetise or distribute broadly, plan a parallel workstream:


    1. Define data requirements: latency, field list, replay availability, redistribution rights, AI usage
       rights.
    2. Evaluate vendors (e.g., Sportradar gRPC live feed) and request sandbox/replay access. 23
    3. Contracting: SLA, usage limits, support channels, compliance.
    4. Integration: implement adapter to canonical schema; validate coverage vs analytics needs.
    5. Go-live readiness: monitoring, failover, replay-based regression tests.

Sportradar’s docs explicitly encourage using replay functionality in integration and describe gRPC
delivery mechanics. 23


Immediate checklist of next steps
     • Choose your latency target explicitly: “internal p95 < 300 ms; user-visible freshness bound by
       OpenF1 cadence (e.g., 4s intervals)” (or decide you need a licensed feed). 42
     • Decide the live MVP provider: OpenF1 MQTT/WSS (requires OAuth2 token flow) vs a licensed
       gRPC feed. 43
     • Confirm your legal posture (non-commercial demo vs commercial) and bake the constraints into
       the product requirements; Formula 1 guidelines explicitly restrict AI/TDM and commercial use
       without licence. 19
     • Build the collector + canonical event log + replay harness first; treat replay as the backbone of
       testing (mirroring the “use replay to develop robust solutions” guidance in commercial feed
       docs). 23
     • Define your “core live analytics primitives” (tower tick, stint model, undercut alert, anomaly flags)
       and how they degrade when telemetry channels vanish. 3
